# Call Detection & Recording — Android Architecture Notes (Expo + React Native)

## 1. Why this needs a native module — it can't be done in JS

React Native / Expo have no JavaScript APIs for telephony state, call logs, call audio, or the Android telecom framework — these are OS-level Android capabilities exposed only through native `android.telephony` and `android.telecom` classes. To use any of them, the JS layer has to call into native code.

This has two practical consequences for the project:

- **Expo Go won't work.** You need a custom dev client (`npx expo prebuild` + `expo-dev-client`, built via EAS Build or locally). This is normal "bare RN with Expo tooling," not a full eject.
- **You need a native module.** Either the modern **Expo Modules API** (`npx create-expo-module --local`) or a classic RN Native Module — written in Kotlin _or_ Java, both fully supported. This module hosts the Android-side logic and emits events (call started, call ended, incoming number, etc.) to JS via an event emitter.

Everything else in this document — call state detection, call log access, recording — is native Android work that this module wraps.

## 2. Core Android building blocks (glossary)

| Concept                               | What it is                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| ------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **`TelephonyManager`**                | System service that exposes phone state (SIM info, network, call state). The foundation for reading telephony state.                                                                                                                                                                                                                                                                                                                                                                                                                                                           |
| **`TelephonyCallback`**               | Modern (API 31+) callback-based replacement for the old `PhoneStateListener`. Used to detect call state transitions: `IDLE` → `RINGING` → `OFFHOOK` → `IDLE`. This is how you detect "a call started" / "a call ended."                                                                                                                                                                                                                                                                                                                                                        |
| **`READ_CALL_LOG` permission**        | Dangerous permission gating read access to Android's system call log (`CallLog.Calls` content provider) — number, timestamp, duration, call type for every call. Runtime-grantable by the user, but see the Play Store distribution restriction below.                                                                                                                                                                                                                                                                                                                         |
| **`RoleManager.ROLE_DIALER`**         | The "default phone app" role. Only one app on the device can hold it. Holding it grants full `InCallService` access (live call control + audio stream) and unlocks `READ_CALL_LOG` for Play Store distribution. **Requires fully implementing your own dial pad UI and in-call UI (incoming + ongoing call screens) — this is a hard requirement, not optional.** If your `InCallService` fails to bind properly, the OS may drop calls or fall back to the preloaded dialer as an emergency measure — it is not a safe way to "get the permission without doing the UI work." |
| **`RoleManager.ROLE_CALL_SCREENING`** | A separate, lighter role for caller-ID/spam apps. Lets you implement a `CallScreeningService`. Does **not** require replacing the default dialer or building any UI.                                                                                                                                                                                                                                                                                                                                                                                                           |
| **`CallScreeningService`**            | The service you implement to hold `ROLE_CALL_SCREENING`. Android calls `onScreenCall(Call.Details)` the moment an **incoming** call rings, before the user sees it — gives you the number in real time. **Does not fire for outgoing calls** — Android removed the old `NEW_OUTGOING_CALL` broadcast in Android 10 for privacy reasons, so there is no equivalent non-privileged signal for outgoing numbers.                                                                                                                                                                  |

## 3. The possible paths, and their limitations

| Path                                                                                   | What you get                                                                                                                                                                                                                                                                                                                                                                                                                                                                                       | Limitations                                                                                                                                                                                                                                                                                                                     |
| -------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **A. `TelephonyCallback` + foreground service only**                                   | Call start/end timestamps, duration. Works for both incoming and outgoing calls.                                                                                                                                                                                                                                                                                                                                                                                                                   | No caller number or name at all. Just "a call happened."                                                                                                                                                                                                                                                                        |
| **B. `ROLE_CALL_SCREENING` + `CallScreeningService`**                                  | Incoming caller's number, in real time, without needing default-dialer status.                                                                                                                                                                                                                                                                                                                                                                                                                     | **Incoming only.** No outgoing number — no OS-level signal exists for non-privileged apps.                                                                                                                                                                                                                                      |
| **C. `READ_CALL_LOG` requested directly (no default handler)**                         | Full call log: numbers, timestamps, duration, direction (incoming/outgoing) — the simplest, most complete option.                                                                                                                                                                                                                                                                                                                                                                                  | **Not distributable on the Play Store** unless you're the default Phone/SMS/Assistant handler, _or_ you qualify for a declared Google Play exception (see path E). Fine for development, sideloading, or internal/enterprise distribution.                                                                                      |
| **D. `ROLE_DIALER` (full default dialer replacement)**                                 | Everything: `READ_CALL_LOG` (incoming + outgoing), live `InCallService` call control, and direct access to the live call audio stream for legitimate recording. Once held, this becomes the _only_ call-handling path on the device — even calls placed via icons/links/contacts route through your app's `ConnectionService`.                                                                                                                                                                     | Big scope: mandatory dial pad UI + full incoming/ongoing call UI (not optional — see glossary above). Once granted, your app is responsible for every call the user makes, including reliably letting them answer/end calls. Emergency calls are always force-routed to the preloaded dialer regardless.                        |
| **E. Google Play "Caller ID, spam detection & blocking" exception**                    | This is the path Truecaller actually appears to use for the majority of its users (who never set it as default dialer). It's a _named, current_ exception category in Google Play's Permissions Declaration Form — apps can request `READ_CALL_LOG` without being the default handler, if their core, documented functionality is genuinely caller ID / spam blocking. Covers incoming **and outgoing** calls, since it grants real system call log access.                                        | Discretionary and reviewed case-by-case — not automatic. Real apps have been rejected under this category when Google's reviewers judged the permission wasn't sufficiently core to the app's stated purpose. Your app's Play Store listing and description must prominently document this as the core feature.                 |
| **F. Ask user to enable device's native call recording + request folder access (SAF)** | A workaround for _recording_ specifically, since direct call-audio recording APIs are closed to third-party apps (blocked since Android 10, and the Accessibility API loophole has since been closed by policy). Uses `expo-file-system`'s `StorageAccessFramework` to get a persisted folder permission, matched against your own `callStarted`/`callEnded` timestamps. Some OEMs also embed the number/contact name in the recording's filename, which can be parsed as a bonus identity signal. | Not universal — not every device/region has native call recording (Pixel and Samsung both region-lock it for legal reasons). Folder paths are OEM/version-specific and require an actively maintained lookup table. Matching by timestamp is a heuristic, not a guarantee. Filename-based identity parsing is best-effort only. |

## 4. Our feasible solution (based on comparing this against how Truecaller actually behaves)

**Phase 1 — current build, not for Play Store distribution**

- `TelephonyCallback` + foreground service (Path A) for call start/end detection.
- `READ_CALL_LOG` requested directly from the user (Path C) — fully functional since we're not going through Play Store review yet.
- SAF folder access for the user's device call recordings (Path F), matched by timestamp; filename parsing as a bonus for identity where the OEM supports it.

**Phase 2 — once ready to publish, without building a full dialer**

- Apply for the **Caller ID, spam detection & blocking** Play Store exception (Path E) — this is the realistic Truecaller-equivalent path. If approved, keeps `READ_CALL_LOG` (incoming + outgoing) without needing `ROLE_DIALER` or any custom in-call UI.
- Layer in `ROLE_CALL_SCREENING` (Path B) for real-time incoming call screening/blocking features, independent of the exception approval.

**Phase 3 — only if the product outgrows Phase 2**

- Full `ROLE_DIALER` replacement (Path D) — undertaken only once the team is ready to build and maintain a complete, reliable dial pad + in-call UI, since this becomes the user's sole calling interface on the device.

This mirrors what the evidence suggests Truecaller itself does: most users interact with it purely through Path E + Path B, and only a minority explicitly opt into full default-dialer mode.

## 5. Note on Java vs. Kotlin

All of the native pieces above (`TelephonyCallback`, `CallScreeningService`, foreground `Service`, `ContentResolver`/`CallLog` queries) are plain Android SDK — fully usable from **Java**, no Kotlin required. Kotlin and Java interoperate directly, so this is a legitimate way to build the module without taking on a new language, even though most current Expo Modules API examples and community RN packages are written Kotlin-first.
